Twin Cities Summary:

Application that retrieves and displays information about two cities from a mySQL database, weather data from OpenWeather API to be displayed with a map, using Google Maps API with points of interests as markers on the map.

The website provides in-depth technical documentation on system design, code implementation and database maintenance.

Documentation follows a logical flow, starting with design following up with implementation.

Before reading the documentation, it's highly recommended to familiarise yourself with the application structure and page layouts, to help understand where the relevant information is stored. This can be viewed in the Github.

Tech Stack:

The technology used includes; PHP, Javascript, HTML & mySQL.

PHP ;

  • Configuration file - holding all data & functions
  • Querying data from Openweather API
  • Inserting HTML to be displayed to user

Javascript;

  • Creating functions & variables to help configure webpage
  • Host Map using Google Maps API

mySQL

  • Holding relevant information about cities
  • Creating tables based off chosen entities and attributes

Models & Design:

The application pulls data from a populated database of twinning cities

To help conceptualise how our data would interact and be stored in the database, we created a conceptual and logical entity-relationship model.
Using Astah & mySQL to help render the models.

The database is populated with information from Edinburgh (Scotland) and Munich (Germany) - these cities both have relatively large populations so they will have sufficient data to build our application from.

Conceptual Model:

The model contains 5 entities;

  • City
  • Places of interest
  • Weather
  • Forecast
  • Images

Each entity has attributes to populate the database with.
For example the 'city' entity has an id, name, lat/lon and population.

Download Conceptual Model

Logical Model:

The logical model transforms the conceptual model into a low-level model to show all Primary & Foreign Keys.

When updating the database, the models will not have to change, unless another entity is appended.

This is because the data populating the database are examples of entities, so to add another place of interest to the database the current model is sufficient.

Configuration File:

The configuration file is used to hold global data and constants including API keys, static data and configuration data

The configuration file can be found here.

There are many methods and functions we can use in the configuration to prevent repetition in code. The config file is imported into all relevant files to share data and constants.

To initialise the configuration file, a namespace and timezone is set.



If a namespace is chosen, it must be declared in the first line. The contents of the file belong to the namespace 'TwinCitiesApplication'.


date_default_timezone_set() sets the default timezone used by all date/time functions. In this case we hard-code it to GMT, however it can be changed depending on location, for example 'date_default_timezone_set("America")'.

Define()

Define() method allows a constant to be declared with a key:value system.
Constants do not change value, so are useful for declaring static values, such as API keys.

OW_API is the key with the API being the value. The same for G_MAPS_API.
The constants 'OW_API' and 'G_MAPS_API' can be used in the API query instead of manually writing the key each time.

$cities array

The $cities variable is being initialised with values such as name, co-ords and API calls from each city, in an array. To add more values or update values to different cities, the values only must be changed once in the configuration file. The 'places[ ]' array holds the name and co-ords of specific points of interest, which can be looped through to add to the map.

Error Handler

The config file also holds an error handler.

The function takes in the error number, error message, file location and the line of the error as arguments in the function. From this it will echo an error message to the user to help identify where the bug has been caused.

Current Weather Data

Forecast Weather Variables

Google Maps API:

Using the Google Maps documentation to initialise and configure our map

Full Google Maps documentation can be found here.

The map is initialised in the cities.php file

Initialising the map

First, we need to initalise the map to add content and append markers to it. A constant 'GoogAPI' is declared in Javascript, to store the API value from the configuration file. 'GoogAPI' is required to allow the API key to be used in Javascript.

The whole map is wrapped in the initMap() function.

We create an object for the map options, to determine it is displayed. During implementation we chose 'zoom:12' , but the zoom can be changed to liking.

The center of the map takes in a JSON encoded version of the places and co-ordinates, to create a dynamic center of the map.

Creating map instance and adding markers

We then create a new map instance, also passing in the options object as an argument to render the map.

The var 'places' uses the JSON encoded version of places.

A 'for' loop is used to loop through all the places to call the addMarker function to add markers onto the map.

The 'var map' selects the div in the HTML with 'id=map' using 'document.getElementById'. The map will be displayed in this divider.

Add markers function

To add markers for our points of interest, a javascript function is used called addMarkers.

Passing in 'place' argument - which will be used to pass in the 'places' variable. (See next section)

Inside the function is a variable 'marker' which will be used for each individual point of interest.

Options are passed into the marker variable using the .Marker method from the Google Maps API.

The 'if (place[3])' checks the validation of content, to ensure all data is correct.

Each marker has an event listener, using '.addListener' to add functionality when a user hovers in and out on a marker, which will then display information about each marker. Upon click, the URL of the website changes to display photos and information about each marker.

Points of interest

The points of interest were chosen by the team, trying to find the most unique places for each city.

The points of interest can be found in the configuration file.

Points of interest are found in the '$placesJson' variable - which calls the $cities array which holds arrays for both Edinburgh and Munich.

The traversal of the $cities array can be done after the right city is retrieved using the $_REQUEST method. After this, places can be found using $cities[$city]['places'];
